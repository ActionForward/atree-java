# Architecture & CI Decisions

This document records the setup decisions behind this project, so they can be reused as a
starting point for other ActionForward Java projects.

The project itself is a plain Java library (the A-Tree expression-matching index — see
`README.md`); it uses the `java-library` and `antlr` Gradle plugins. The ANTLR tool is kept off
the published api/runtime classpath (only `antlr4-runtime` is an `implementation` dependency —
see the `configurations.api` block in `build.gradle.kts`).

## Toolchain: Devbox

[Devbox](https://www.jetify.com/devbox) provisions the JDK and Gradle via Nix, so neither needs
to be installed on the host (dev machine or CI runner). Everything runs via `devbox run <script>`
or inside `devbox shell`.

Current pins (`devbox.json`):
- JDK: `javaPackages.compiler.temurin-bin.jdk-25@latest` (Eclipse Temurin 25)
- Gradle: `gradle_9@latest` (9.x)

Use [nixhub.io](https://www.nixhub.io) to look up available package names/versions before
changing a pin in `devbox.json`.

`gradle.properties`'s `org.gradle.java.home` line is managed by devbox — it's rewritten to match
whatever JDK devbox resolves. If it ever looks stale, delete the line (or run `devbox install`)
to regenerate it.

The self-hosted runner (`netcup`) has `devbox` installed once, by root. `ci.yml` assumes it's
already on `PATH` and never installs it per job — simpler and faster than reinstalling the
toolchain on every run.

`devbox.json` also defines `build`/`test` scripts on purpose:

```json
"build": "gradle assemble testClasses",
"test": "gradle test"
```

A plain `gradle build` already runs tests (via `check`). Splitting it this way keeps local
commands granular — compile-only vs. run-tests-only. CI still merges them into one Gradle
invocation (`gradle build`) for speed, since that distinction doesn't matter there.

## Gradle configuration: bounded RAM, fast CI regardless

The runner has only 2 GB RAM, so every Gradle-related setting below keeps memory usage bounded
without sacrificing CI speed:

| Setting | Where | Why |
|---|---|---|
| `org.gradle.jvmargs=-Xmx512m` | `gradle.properties` | Caps the build JVM's heap. |
| `org.gradle.workers.max=2` | `gradle.properties` | Limits concurrent Gradle worker processes. |
| `maxParallelForks = 1`, `maxHeapSize = "384m"` | `build.gradle.kts` `Test` task | Caps the separate forked test JVM. |
| `--no-daemon` (CI only) | `ci.yml` | Avoids a long-lived background JVM piling up across CI runs on a persistent runner. |
| `org.gradle.caching=true` + `org.gradle.configuration-cache=true` | `gradle.properties` | Both are disk-based and daemon-independent, so they claw back the speed `--no-daemon` gives up — unchanged tasks are skipped instead of re-run. |

Note: `--no-daemon` only works reliably when passed as `devbox run -- gradle build --no-daemon`
(the raw-command form), not as a trailing flag after a named script — that's why `ci.yml`
invokes Gradle directly.

## CI pipeline (`.github/workflows/ci.yml`)

- **Runner**: always `[self-hosted, netcup]` — fixed, not something to parameterize per project.
- **Checkout**: `clean: false`, so the configuration cache (stored in the gitignored, project-local
  `.gradle/` folder) survives between runs on this persistent runner.
- **One "Build & run tests" step**: `devbox run -- gradle build --no-daemon` — compiles, packages,
  and tests in a single Gradle invocation/JVM startup.
- **Test reporting**: [`EnricoMi/publish-unit-test-result-action`](https://github.com/EnricoMi/publish-unit-test-result-action),
  chosen for richer rendering (PR comments, inline annotations) than simpler alternatives. Uses
  the `/linux@v2` variant (Docker container action; the `composite` variant is deprecated), so the
  runner's user needs to be in the `docker` group.
- **Permissions**: `contents: read`, `issues: read`, `checks: write`, `pull-requests: write` — the
  superset needed for a private repo, used unconditionally so the same `ci.yml` works whether a
  given project's repo is public or private.
- **Badge**: a pass/fail SVG badge generated by `emibcn/badge-action` and pushed to a GitHub Gist
  via `actions/github-script` (official, Node 24) calling the Gist API directly, rather than a
  dedicated third-party "append to gist" action — that niche had nothing well-maintained. Runs
  only on pushes to `main`, and needs a `GIST_TOKEN` repo secret (a personal access token with
  `gist` scope).
- Third-party actions are pinned to a version tag (e.g. `emibcn/badge-action@v2.0.4`).
- All GitHub Actions dependencies run on Node 24, to avoid GitHub's Node 20 deprecation warning.

## Using this repo as a template for new projects

This repo is marked as a **GitHub template repository**. To start a new ActionForward Java
project from it: on GitHub, go to the repo page → **Use this template** → **Create a new
repository**, pick the new repo's name/visibility under the `ActionForward` org, and clone it.

**Edit these — the per-project checklist:**
1. `devbox.json` → `packages`: JDK version pin and Gradle major version pin (look up names/versions
   on [nixhub.io](https://www.nixhub.io)).
2. `build.gradle.kts` → `java.toolchain.languageVersion`: **must match** the JDK version chosen
   in step 1.
3. `build.gradle.kts` → plugins and dependency versions (for this project: the `antlr`
   tool/runtime pair; for a Spring Boot project: `id("org.springframework.boot") version "..."`).
4. `build.gradle.kts` → `group`/`version`, `settings.gradle.kts` → `rootProject.name`, and the
   `com.actionforward.atree` package path under `src/` — rename to match the new project.
5. If you changed the JDK version, delete `gradle.properties`'s `org.gradle.java.home` line —
   devbox regenerates it correctly on the next `devbox run`.
6. For the badge: create a **new** Gist for this project (`gh gist create`), swap its ID into
   `ci.yml`'s "Upload badge to Gist" step and into the badge image URL in this project's
   `README.md`, and add a `GIST_TOKEN` repo secret.

**Leave fixed — copy as-is, don't reinvent per project:**
- `runs-on: [self-hosted, netcup]` and the entire CI pipeline structure/steps
- The RAM-bounding `gradle.properties` settings and the `Test` task caps in `build.gradle.kts`
- `checkout`'s `clean: false` and the `--no-daemon` raw-command invocation pattern
- The general shape of the test-reporter + badge steps (only the Gist ID/secret changes per
  project, per item 6 above)
